# Security Threat Model

This threat model applies to Report Lab and every future SSAMENJ project from day one. Security is a product requirement, not a launch hardening phase.

## Threat Categories

| Threat category | What the attacker tries to do | Where it applies | Required default controls | Required tests |
| --- | --- | --- | --- | --- |
| Opportunistic scanners and bots | Discover exposed endpoints, debug pages, weak headers, and known vulnerable versions. | Public frontend, Express API, `/health`, `/api/*`, public report and Smart Pages links. | Security headers, CORS allowlist, route inventory, no debug routes in production, rate limits, safe health responses. | Public endpoints do not leak env values; unknown routes do not expose internals; API rejects abusive request volume. |
| Credential stuffing and brute force | Reuse leaked passwords or rapidly guess credentials. | Login, password reset, creator auth, staff access, platform admin access. | Rate limit login/reset, strong password hashing, session/token versioning, MFA for internal/admin systems, safe login errors. | Login rate limit; invalid/expired token blocked; deactivated user blocked; tokenVersion mismatch blocked. |
| Broken access control | Use a valid session for actions beyond the user's role. | Students, imports, reports, settings, NFC gate/canteen/wallets, Smart Pages billing/templates. | `requireAuth`, `requirePermission`, server-side permission checks, route-level tests, no frontend-only authorization. | Wrong role blocked; wrong permission blocked; hidden button is not treated as security. |
| Tenant isolation bypass | Read or mutate another school/company's data. | Every table with `schoolId`, public tokens, AI context, import batches, NFC tags/wallets, report releases. | `resolveTenantContext`, token tenant wins, body/query tenant cannot override token tenant, `schoolId` on all tenant queries, cross-tenant 403. | Tenant A cannot read/write Tenant B; body tenantId override ignored; missing tenant in production returns 401; mismatch returns 403. |
| Injection attacks | Execute unintended database, shell, script, template, or prompt behavior. | Query params, JSON bodies, imports, report comments, Smart Pages document data, AI prompts. | Zod validation, Prisma parameterization, output encoding, no shell execution from user data, AI data/instruction separation. | Validation rejects malformed input; raw provider/Prisma errors hidden; formula injection escaped in exports. |
| File upload attacks | Upload malware, oversized files, parser bombs, spoofed files, or path traversal names. | Passport photos, settings assets, scan uploads, Smart Pages source files, student imports. | Size limits, file count limits, zero-byte rejection, MIME plus extension validation, signature checks, filename sanitization, isolated storage. | Empty file rejected; invalid MIME rejected; invalid extension rejected; oversized upload rejected; PDF/XLSX signatures checked. |
| Import/parser attacks | Poison data through CSV/XLSX/PDF/OCR rows, formulas, invalid encodings, oversized matrices, or duplicate rows. | Marks import, student import, scan import, Smart Pages extraction, error CSV export. | Shape limits, dry-run before commit, row errors, formula escaping, atomic commits, audit log on commit/failure. | Dry-run does not commit; commit writes audit log; row errors preserved; CSV formula injection blocked. |
| API abuse and scraping | Enumerate students, reports, public links, verification tokens, or billing state. | Public report pages, verification endpoints, search endpoints, list endpoints. | Pagination, search limits, token entropy, rate limits, audit suspicious access, avoid sequential public IDs. | Public token guessing blocked; pagination limits enforced; scraping volume triggers rate limit/security event. |
| Insider threats | Abuse legitimate access to change data, export records, delete evidence, or deploy unsafe code. | Admin console, production DB, deployment pipeline, audit logs, platform-owner routes. | Least privilege, dual authorization for high-risk changes, audit logs, immutable backups/logs, production access reviews. | Audit log required for sensitive action; destructive action requires reason/confirmation; admin-only path blocked for non-admin. |
| Social engineering | Trick operators or developers into revealing secrets, approving unsafe changes, or running malicious instructions. | Support workflows, WhatsApp/contact links, email, Codex/agent instructions, admin operations. | Hardware MFA for internal platforms, approval workflows, out-of-band verification for sensitive actions, agent instruction hygiene. | Sensitive action requires confirmation; support/admin docs include verification steps; AI instructions cannot override repo rules. |
| Supply-chain attacks | Introduce compromised packages, scripts, models, or generated code. | npm dependencies, Prisma scripts, build tooling, OCR/AI SDKs, Vercel/Railway deploys. | Lockfiles, dependency review, minimal scripts, signed artifacts where possible, no postinstall trust for production secrets. | Build uses lockfile; dependency changes reviewed; scripts cannot read secrets unnecessarily. |
| Infrastructure/env/secret leakage | Expose DB URLs, JWT secrets, API keys, platform tokens, or upload credentials. | `.env`, logs, health endpoints, Vercel/Railway envs, frontend env vars. | Server-only secret naming, `.env.example` without values, redaction, health endpoints return SET/MISSING only. | Health endpoint does not leak secret values; logs redact tokens; frontend bundle contains no server secret names/values. |
| Misconfiguration | Ship permissive CORS, debug auth fallback, unsafe upload storage, missing migrations, or wrong public URL. | Express server, deployment config, env vars, storage provider, CORS. | Env validation, production-mode checks, explicit CORS allowlist, migration deploy, upload storage validation. | Production missing env fails startup; CORS rejects unknown origins; upload storage misconfig returns safe 503. |
| Ransomware/data destruction | Delete, encrypt, corrupt, or mass-update tenant data. | Database, uploads, public assets, import commits, admin/bulk actions. | Backups, least privilege DB users, reason fields, soft-delete where possible, audit logs, dual approval for destructive operations. | Bulk/destructive action requires confirmation and audit; unauthorized delete blocked; restore procedure documented. |
| Audit-log tampering | Remove evidence or hide malicious actions. | `AuditLog`, import job summaries, platform logs, deployment logs. | Append-only audit service, restricted update/delete access, external log drain/WORM storage where possible, correlation IDs. | Sensitive actions create audit entries; non-admin cannot mutate audit records; audit event includes actor and tenant. |
| Payment/wallet fraud | Steal funds, fake top-ups, bypass PIN checks, reverse charges without authority, or replay transactions. | NFC wallet, canteen, top-up, reconciliation, fee holds. | Permission checks, PIN rules, idempotency keys, audit logs, reconciliation approval, reversal reason. | Wrong role blocked; duplicate idempotency blocked; reversal requires reason; reconciliation approval restricted. |
| Public verification abuse | Scrape or brute force verification links, parent report tokens, Smart Pages public URLs, or NFC tag links. | `/verify`, parent reports, published Smart Pages, NFC public tag routes. | High-entropy tokens, rate limits, token expiry where appropriate, no tenant leakage on not-found, audit suspicious access. | Invalid token returns safe response; brute-force attempts rate-limited; public route does not reveal tenant existence. |

## Mandatory Design Rule

Every new route, job, upload, import, public token, and AI feature must explicitly define auth, permission, tenant scope, validation, rate-limit need, audit need, and tests before implementation.
