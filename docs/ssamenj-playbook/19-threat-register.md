# Threat Register

This register is the reusable baseline for SSAMENJ projects. Status values should be updated per project as controls are implemented and verified.

| Threat ID | Threat category | Attack example | Impact | Likelihood | Required controls | Required tests | Status |
| --- | --- | --- | --- | --- | --- | --- | --- |
| HUMAN-001 | Insider threat | Admin exports data, changes records, or hides traces. | High | Medium | Least privilege, auditService, dual approval for destructive actions, immutable backups/logs. | Sensitive action writes audit; wrong role blocked; destructive action requires reason. | Baseline documented |
| HUMAN-002 | Social engineering | Deepfake or spoofed message asks staff to approve access/payment. | High | Medium | Hardware MFA for internal systems, confirmation, out-of-band verification, securityEventService. | Sensitive flow requires confirmation; support/admin action verified. | Baseline documented |
| WEB-001 | Opportunistic scanning | Bot probes `/api`, debug routes, old assets, and health endpoints. | Medium | High | Security headers, safe health responses, route inventory, rate limits. | Health does not leak env; unknown route safe; abuse rate-limited. | Baseline documented |
| WEB-002 | Credential stuffing | Attacker reuses leaked passwords against login. | High | High | Rate limits, strong hashing, tokenVersion, safe login errors, MFA for internal systems. | Login rate limit; expired/deactivated token blocked. | Baseline documented |
| AUTHZ-001 | Broken access control | Teacher calls admin-only student/import/report route. | High | Medium | requireAuth, requirePermission, permission tests, no frontend-only checks. | Wrong role/permission blocked. | Partially implemented in Report Lab |
| TENANT-001 | Tenant isolation bypass | School A loads School B report, import batch, vector, wallet, or tag. | Critical | Medium | resolveTenantContext, token tenant wins, schoolId filters, cross-tenant 403. | Tenant A cannot read/write Tenant B; body tenantId override blocked. | Partially implemented in Report Lab |
| UPLOAD-001 | Unsafe file upload | Spoofed PDF/XLSX/image, malware, oversized file, zero-byte file. | High | High | uploadSafety, fileSignature, size/count limits, safe storage. | Empty/invalid/oversized/signature tests. | Partially implemented in Report Lab |
| IMPORT-001 | Parser/import poisoning | CSV formula, oversized matrix, duplicate rows, invalid scores commit. | High | Medium | Spreadsheet limits, formula escaping, dry-run, row errors, audit logs. | Dry-run no commit; formula injection escaped; commit audit exists. | Partially implemented in Report Lab |
| API-001 | API abuse/scraping | Enumerate public report tokens or verification links. | Medium | Medium | High-entropy tokens, rate limits, pagination limits, safe not-found. | Token guessing blocked; public route rate-limited. | Baseline documented |
| AI-001 | Direct prompt injection | User asks AI to ignore permissions and reveal reports. | High | High | aiSafetyPolicy, backend permissions, aiOutputValidator, no direct DB writes. | Injection cannot bypass permission; unsafe output rejected. | Baseline documented |
| AI-002 | Indirect prompt injection | Uploaded PDF/OCR text contains hidden tool instructions. | High | High | Data/instruction separation, document safety scan, aiToolGuard. | Uploaded prompt treated as data; tool call blocked. | Baseline documented |
| RAG-001 | RAG poisoning | Malicious document dominates retrieval or changes decisions. | High | Medium | ragIngestionPolicy, provenance, chunk limits, quarantine/reindex. | Poisoned doc removable; source logging present. | Baseline documented |
| RAG-002 | Cross-tenant AI context leak | Same file hash/vector cache leaks between schools. | Critical | Medium | Tenant-scoped cache/index queries, source ownership checks, aiAuditLog. | Same hash in another tenant not reused; retrieval tenant-scoped. | Partially implemented in Report Lab |
| AGENT-001 | Agent tool abuse | AI tool bulk releases reports, changes wallet balance, or deletes data. | Critical | Medium | aiToolGuard, least-privilege tools, confirmation, auditService. | Sensitive tool requires permission and confirmation. | Baseline documented |
| SUPPLY-001 | Supply-chain attack | Compromised npm package, model file, or remote prompt. | High | Medium | Lockfiles, dependency review, signed datasets/prompts where possible, provider allowlist. | Build uses lockfile; unknown provider/prompt rejected. | Baseline documented |
| INFRA-001 | Secret/env leakage | JWT secret, DB URL, API key exposed in logs, frontend, or health route. | Critical | Medium | redactSecrets, safe health, server-only env, env validation. | Health returns SET/MISSING; frontend bundle has no server secrets. | Partially implemented in Report Lab |
| INFRA-002 | Misconfiguration | Permissive CORS, missing storage env, debug fallback in production. | High | Medium | Env validation, CORS allowlist, production guards, deployment checklist. | Unknown CORS origin blocked; production missing env fails. | Partially implemented in Report Lab |
| DATA-001 | Ransomware/data destruction | Bulk delete/import corrupts tenant records. | Critical | Low-Medium | Backups, audit logs, reason fields, dual approval for destructive ops. | Destructive action requires reason/audit; restore documented. | Baseline documented |
| AUDIT-001 | Audit-log tampering | Attacker deletes or edits audit evidence. | High | Medium | Append-only auditService, restricted DB access, external log drain/WORM storage. | Audit records cannot be mutated by app roles. | Baseline documented |
| PAY-001 | Wallet/payment fraud | Fake top-up, duplicate transaction, unauthorized reversal. | Critical | Medium | Permission checks, idempotency, PIN rules, reconciliation approval, audit. | Duplicate idempotency blocked; reversal requires reason. | Partially implemented in Report Lab where NFC applies |
